01 // The operating system

Agile speed.
Secure rails.

Every mission moves through the same reviewable path. Specialist roles propose; the deterministic engine validates transitions; humans decide at the plan and release gates by default.

  1. 01

    Frame

    Intake

    Turn a rough request into scope, constraints, acceptance criteria, and a definition of done.

    requirements.md
  2. 02

    Defend

    Threat model

    Map assets, trust boundaries, abuse paths, and the controls the implementation must preserve.

    threat-model.md
  3. 03

    Sequence

    Plan

    Split the work into small, ordered stories with owners, dependencies, and explicit gates.

    backlog.md
  4. 04

    Execute

    Build

    Produce typed, path-bounded file proposals; write them only after an explicit apply run and approved plan.

    implementation.md
  5. 05

    Challenge

    Verify

    Run configured checks plus independent code, test, and security review against the acceptance criteria.

    verification/
  6. 06

    Decide

    Release

    Package the change, summarize risk, and pause for the approvals your policy requires.

    release-notes.md

06 // Debrief

Questions
before launch.

Still evaluating? Open an issue on GitHub. The runbook gets better in public.

Ask the maintainers ↗
Is ForgeWard just another coding agent?

No. It is a coordination and control layer for multiple specialist roles. Models do the reasoning; ForgeWard defines who acts, what context they receive, which artifact they owe, and which gate comes next.

Does it require OpenAI?

No. The built-in HTTP adapter follows the familiar OpenAI request and response shape, so compatible cloud gateways and local runtimes can slot in. Install the optional LiteLLM extra for native provider dialects; use a separate HTTPS LiteLLM gateway when you need ForgeWard's stricter transport and body-limit boundary.

Is the adapter layer like an ORM for models?

That is the idea: ForgeWard owns stable provider-neutral request/result types, then uses an OpenAI-compatible HTTP driver or optional LiteLLM driver underneath. Capabilities stay explicit instead of being silently flattened.

Can agents change my code without approval?

Only after you start with --apply and approve the plan gate. ForgeWard then restricts paths, commands, and check-process environment variables, and holds release for a human by default. Native runs cannot deny network access; use an external sandbox when that boundary matters.

Where do prompts, code, and credentials go?

Code and run artifacts stay in your workspace. Model-bound context goes only to the endpoint you configure. Credentials are referenced by environment-variable name and should never be written into prompts or project files.

Is it ready for production?

ForgeWard is an early, forkable field kit. Use it in a sandbox, inspect the policies, and keep humans on consequential approvals while the project hardens through real-world use.
